Here's a complete, self-contained setup — the zip drops straight into your project root as a .devcontainer/ folder. Unzip it next to your pyproject.toml so the layout is:
graph-context-mcp/
├── pyproject.toml
├── src/...
└── .devcontainer/
    ├── devcontainer.json
    ├── docker-compose.yml
    ├── Dockerfile
    └── init-firewall.sh
Why you couldn't find the files last time: when you use a prebuilt image or a "Features"-based config, VS Code generates the actual container definition internally and stores it under its own data directory, so there's nothing obvious to run by hand. The fix is exactly what's done here: the container is defined by a plain docker-compose.yml, and devcontainer.json just points at that compose service. Docker owns the container; VS Code merely attaches to it.
How to run it independently (from a WSL or PowerShell terminal in the project root):
bashdocker compose -f .devcontainer/docker-compose.yml up -d --build
That builds the image, starts the container, runs the firewall lockdown, and keeps it alive. You can shell in with docker exec -it graph-context-mcp-dev bash, and stop everything with docker compose -f .devcontainer/docker-compose.yml down. To connect VS Code afterward, open the project folder and run Dev Containers: Reopen in Container — because the devcontainer.json references the same compose file, VS Code reuses the already-running container instead of creating a parallel one (alternatively, Dev Containers: Attach to Running Container works too). First attach takes a minute while VS Code installs its server and the Python/Ruff/mypy/Claude Code extensions inside.
What's in the box: Python 3.11 (matching your requires-python), git, Node 20 + Claude Code (run claude in the container terminal once to log in; the login persists in a named volume across rebuilds), and hatchling/pytest/ruff/mypy pre-baked into the image so pip install -e . --no-build-isolation works without ever touching PyPI at runtime.
The lockdown is a default-deny iptables/ipset egress firewall adapted from Anthropic's reference dev container, which includes an init-firewall.sh script that blocks all outbound traffic except the domains Claude Code and your development tools need. The allowed paths are exactly the four you asked for: GitHub (via their officially published IP ranges, covering both HTTPS and SSH git), the Anthropic/Claude endpoints plus the npm registry for CLI self-updates, the VS Code server-download and marketplace domains (without these, attaching from the host fails because the extension downloads its server through the container's network), and the Docker host on ports 31009/31012 only — nothing else on your machine is reachable. Everything else gets an immediate REJECT, and the script self-verifies on startup (it fails loudly if example.com is reachable or GitHub isn't). Claude
About Anytype: Anytype is a desktop app, and its API ships directly with the desktop app, running entirely on localhost — by default, Anytype listens on port 31009. So the flow is: Anytype runs on your Windows host as usual, and the container reaches it at http://host.docker.internal:31009 (already set as ANYTYPE_API_BASE_URL in the compose file; use port 31012 instead if you run anytype-cli). You'll need to approve an app key from inside Anytype the first time your code authenticates. WP14 (ADR 019) prepares a headless anytype-cli SIDECAR (compose service `anytype`, opt-in via --profile sidecar): a bot account with its own outbound sync and no rate limit, reachable at http://anytype:31012 over the compose network; at cutover ANYTYPE_API_BASE_URL flips to it and the desktop app becomes a human-only surface. Going the other direction, your own MCP server's port 8000 is published as 127.0.0.1:8000 on the host — bound to loopback only, so nothing on your network can reach it. If your server uses a different port, change it in both the compose ports: and the firewall's --dport 8000 line. Anytype + 2
Two small practical notes: when attached through VS Code, your git credentials and SSH agent are forwarded automatically; when working purely standalone, either use HTTPS with a token or add a read-only mount of your SSH key to the compose file. And if some tool inside the container ever can't connect, the firewall script is the first place to look — add the domain to the ALLOWED_DOMAINS array and re-run sudo /usr/local/bin/init-firewall.sh.
